# Свободен — Android-приложение для отказа от вредных привычек

## Как открыть проект
1. Откройте Android Studio (Ladybug/Koala или новее — нужен AGP 8.7+, Kotlin 2.0+).
2. File → Open → выберите папку `svoboden` (корень с `settings.gradle.kts`).
3. Дождитесь Gradle Sync.
4. Запустите конфигурацию `app` на эмуляторе/устройстве с API 26+.

## Что реализовано
- MVP: онбординг, панель прогресса, журнал, статистика (упрощённая — список по дням), настройки.
- Push-уведомления (WorkManager + Hilt).
- Настройка единиц/целей на привычку.
- Импорт/экспорт JSON-бэкапа.
- Виджет на домашний экран (Glance).
- Достижения/бейджи.
- Биометрическая блокировка приложения.
- Шифрование БД (SQLCipher).
- Семейный режим (несколько профилей).
- Единый splash-flow (AppInitializer) и сводный файл миграций (Migrations.kt).

## Известные упрощения / TODO (см. также FIXES.md)
- Vico-график заменён на упрощённый список по дням — снижает риск несовпадения API версии Vico 2.0 с примерным кодом. Зависимость `vico-compose-m3` оставлена в Gradle для тех, кто захочет добавить график.
- Иконка приложения — простая vector-заглушка (`@drawable/ic_launcher`), замените на реальный ассет.
- Экспорт/импорт работает только с активным профилем (не выгружает все семейные профили разом).
- PIN-хеширование профиля использует статичную соль — для продакшена нужна per-profile случайная соль.
- ReminderWorker использует системную иконку-заглушку `android.R.drawable.ic_dialog_info` — замените на свою.

# Отчёт о проверке проекта — найденные и исправленные ошибки

Проект собирался поэтапно на протяжении 10+ сообщений диалога. При сборке
реальных файлов и статической проверке согласованности обнаружены и
исправлены следующие проблемы.

## Критические (не собралось бы / падало в рантайме)

1. **Отсутствовал плагин kotlinx.serialization** в `build.gradle.kts`, хотя
   `ExportDataUseCase`/`ImportDataUseCase` использовали `kotlinx.serialization.json`
   с самого начала. Добавлен `alias(libs.plugins.kotlin.serialization)`.

2. **Несуществующий класс `SupportOpenHelperFactory`** в `DatabaseModule` —
   реальный класс библиотеки `net.zetetic:android-database-sqlcipher` называется
   `net.sqlcipher.database.SupportFactory`. Также метод `rawExecSQL` не существует
   в SQLCipher API — заменён на `execSQL`.

3. **Новые привычки получали `id = 0` у всех сразу.** `HabitDao.insertAll()` не
   возвращал сгенерированные id, поэтому `SaveHabitsUseCase` создавал сессии
   (`StartSession`) по `habit.id = 0` для всех одновременно выбранных на
   онбординге привычек — ломало FK и путало стрики между привычками.
   Исправлено: `insertAll()` теперь возвращает `List<Long>`,
   `HabitRepositoryImpl.saveHabits()` возвращает привычки с реальными id.

4. **Критический баг инициализации профиля.** На чистой установке таблица
   `profiles` пуста. `ActiveProfileHolder.activeProfileId` оставался `null`
   навсегда, а `HabitRepositoryImpl.saveHabits()` требовал ненулевой активный
   профиль и падал с `error(...)` прямо на экране онбординга. Исправлено:
   `ActiveProfileHolder.init()` теперь создаёт профиль по умолчанию, если
   профилей ещё нет.

5. **Как следствие фикса №4** — старая логика выбора стартового экрана
   (`profileRepository.count() == 0 → Onboarding`) сломалась бы, потому что
   после автосоздания профиля `count()` уже никогда не равен 0. Исправлено:
   стартовый экран теперь определяется флагом `UserProfile.onboardingCompleted`,
   а `count()` используется только для решения «показывать ли выбор профиля»
   (2+ профиля — семейный режим).

6. **`ImportDataUseCase` вставлял записи журнала по старым `habitId`**
   из импортированного файла, хотя привычки при импорте получают новые id от
   Room (autoincrement) — вставка упала бы на FK-constraint. Исправлено:
   строится карта `oldHabitId → newHabitId`, записи переписываются перед вставкой.

7. **`SessionRepository` и `ProfileRepository` как domain-интерфейсы
   никогда не были формализованы** — упоминались и использовались в
   use case'ах (`CalculateStreakUseCase`, `LogJournalEntryUseCase`,
   `GetBestStreakUseCase`, `ProfileSelectViewModel` и др.) на протяжении
   нескольких сообщений, но объявлены не были. Без них ничего не
   скомпилировалось бы. Добавлены интерфейсы и реализации.

8. **`WorkManagerModule` из чернового плана был синтаксически неполным**
   (содержал `...` вместо кода) и в принципе не нужен: `androidx.hilt:hilt-work`
   сам генерирует `HiltWorkerFactory` через `@HiltWorker` + `@AssistedInject`.
   Модуль не создавался; вместо этого добавлен `ksp(libs.hilt.work.compiler)`
   в зависимости — без него аннотация `@HiltWorker` не сработает.

9. **`AppPreferences.reminderTime` был описан как `Flow<Pair<Int, Int>>`
   напрямую из DataStore** — Preferences DataStore не умеет хранить `Pair`.
   Час и минута хранятся раздельными ключами и объединяются через `combine()`.

10. **`accompanist-flowlayout` использовался в примерах Compose-экранов**,
    хотя такой зависимости нет в Gradle. Заменено на нативный
    `androidx.compose.foundation.layout.FlowRow` (входит в Compose Foundation
    с версии 1.6+, доступен без доп. зависимостей).

## Существенные (логические баги, не ломающие сборку)

11. **`Habit.isForever` был constructor-параметром с дефолтом**
    (`val isForever: Boolean = goalDays == null`), а не вычисляемым свойством —
    при `copy(goalDays = X)` без явного обновления `isForever` поля могли
    рассинхронизироваться. Исправлено: `isForever` теперь `get()`-свойство,
    невозможно проставить неправильное значение вручную.

12. **`UserProfileRepositoryImpl.completeOnboarding()` затирал `motivation`
    пустой строкой** вместо сохранения текущего значения. Исправлено: метод
    читает текущее состояние и меняет только флаг `onboardingCompleted`.

13. **Денормализация `profileId` в 4 таблицах вместо 1.** Черновой план главы 9
    добавлял `profileId` в `habits`, `journal_entries`, `sessions` и
    `unlocked_achievements` одновременно — риск рассинхронизации при каждой
    вставке. Упрощено: `profileId` есть только в `habits`; остальные таблицы
    масштабируются на профиль транзитивно через `habitId → habits.profileId`.

14. **`JournalEntry.profileId` в доменной модели всегда был равен 0**
    (мёртвое поле, не отражённое в Entity после фикса №13). Поле удалено.

## Мелкие

15. `xmlns:tools` был объявлен на дочернем XML-элементе вместо корневого
    `<manifest>` — невалидный XML. Перенесено на корень.
16. Ссылки на `@mipmap/ic_launcher` заменены на `@drawable/ic_launcher`
    (vector-заглушка) — не требует набора PNG по плотностям экрана.
17. Неиспользуемый импорт `android.security.keystore.KeyProperties` в
    `DatabaseKeyProvider` убран.

## Осознанные упрощения (не баги, но стоит знать)

- Vico-график (глава про статистику) заменён на простой список по дням —
  сокращает риск несовпадения версии API Vico 2.0 с примерным кодом из
  диалога. Зависимость оставлена в Gradle для желающих добавить график.
- PIN-хеширование профиля использует статичную соль (см. комментарий в
  `ProfilePinHasher.kt`) — упрощение для MVP.
- Иконка приложения и иконка уведомлений — заглушки, требуют замены на
  реальные ассеты перед публикацией.

## Что НЕ проверено (нет доступа к Android SDK/Gradle в этой среде)

Сеть в этой среде не имеет доступа к `dl.google.com`/Maven Central для
Android-зависимостей, поэтому **фактическая сборка через Gradle не
запускалась**. Все проверки — статические (соответствие пакетов и
директорий, соответствие интерфейсов и реализаций, разрешение внутренних
импортов на реально объявленные классы). Рекомендуется при первом открытии
в Android Studio сразу запустить полную пересборку (Build → Rebuild Project)
и обратить внимание на Logcat при первом запуске — миграция шифрования
БД (глава 8) — самое хрупкое место рантайма, которое нельзя проверить статически.

## Доработка: полноценный Material Design 3

Изначальная тема использовала Material 3 (`androidx.compose.material3`), но
только частично — 5 цветов вместо полной палитры ролей, пустая типографика,
не было shapes-системы и Material You. Доработано:

1. **Полная ColorScheme** (`Color.kt`) — добавлены все обязательные Material 3
   роли (`error`, `outline`, `surfaceVariant`, `tertiary` и парные `on*`),
   и для светлой, и для тёмной темы. Без этого стандартные компоненты
   (`OutlinedTextField`, `HorizontalDivider`, disabled-состояния) подставляли
   дефолтные Material-фиолетовые тона, выбивавшиеся из зелёной айдентики.
2. **Полная типографика** (`Type.kt`) — все 15 стилей шкалы Material 3
   (display/headline/title/body/label × large/medium/small) вместо
   пустого `Typography()`.
3. **Шкала форм** (`Shape.kt`, новый файл) — единый источник скруглений
   вместо литералов `RoundedCornerShape(16.dp)`, разбросанных по каждому
   экрану отдельно.
4. **Material You** (динамические цвета из обоев, Android 12+) — добавлен
   параметр `useDynamicColor` в `SvobodenTheme`, тумблер в настройках,
   ключ в `AppPreferences`. Выключен по умолчанию — авторская зелёная
   палитра важна для айдентики приложения, включается по желанию пользователя.
5. **FAB на Dashboard** — стандартный Material-паттерн для главного действия
   экрана («Отметить день»), раньше приходилось идти через нижнюю навигацию.
6. **Material Motion** — переходы между экранами (`AppNavGraph`) получили
   fade+slide анимацию вместо мгновенной подмены.
7. **Фикс попутно найденного бага**: 3 из 4 вкладок нижней навигации на
   Dashboard использовали одну и ту же иконку `CalendarMonth` — визуально
   неразличимы. Заменены на `Home`/`EditCalendar`/`BarChart`/`EmojiEvents`.
8. **MainActivity теперь реально применяет тему пользователя** — раньше
   `SvobodenTheme` вызывалась без аргументов (дефолт `SYSTEM`, без Material
   You) даже после того, как пользователь менял тему в настройках. Теперь
   `ReadyAppContent` читает `UserProfile.themeMode` и `dynamicColorEnabled`
   и переприменяет тему вложенным вызовом `SvobodenTheme(...)`.
